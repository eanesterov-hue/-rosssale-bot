
Product Mind Map: Cross-Sales внутри брокеров (Whitewill)

0. Контекст и цель (из транскрипции)
	•	Боль: процесс “объект → найти клиентов/брокеров” сейчас вручную (чаты/память), нет системной подсветки.
	•	Ценность: автоматическая подсказка “у кого есть подходящие клиенты”, чтобы брокер по объекту мог быстро связаться внутри компании.
	•	Позиционирование эффекта: увеличение показов → рост сделок.
	•	Ограничение: данных для точного мэтчинга мало/они неструктурированы (комментарии, звонки, WhatsApp/Telegram).

⸻

1) MVP линия (самый высокий приоритет — то, что прям проговорили на встрече)

1.1 V1 — “Объект/ЖК → список брокеров по показам”

Почему приоритет №1 (из транскрипции):
	•	“Самое простое — обратиться к показам”, “первая версия, чтобы протестировать”.
	•	Данные уже есть: Саша вручную фильтрует показы и выгружает в Excel.

Функция:
	•	Вход: объект/ЖК, период (7/14/30).
	•	Выход: уникальный список имён брокеров, у которых были показы по этому объекту/ЖК.

Цель V1: быстро подтвердить полезность и запустить процесс “объект → внутренний контакт брокеров”.

⸻

2) Улучшения внутри той же механики показов (следующий приоритет, тоже звучал)

2.1 V1.1 — “Нормализация объектов / алиасы”
	•	“Прайм парк” vs “Прайм парк вторичка”, разные написания.
	•	Решение: exact/contains + словарь алиасов.

2.2 V1.2 — “Мини-фильтры по статусу/периоду”
	•	Запланирован/Завершен/Отменен.
	•	Периоды: по умолчанию 14 дней, настройка.

2.3 V1.3 — “Вывод готовый под интерфейс”
	•	JSON + текст (без метрик, просто список имён).

⸻

3) V2 — точность и исключения (прямо проговорено как развитие)

3.1 V2 — “Не предлагать тех, кто уже видел этот объект”

Почему:
	•	В транскрипции: “вторая версия — исключение, не показывай тех, кто моего собственника уже видел”.

Функция:
	•	Вход: объект/ЖК + конкретный лот/собственник (если появится ID).
	•	Логика: исключить брокеров/клиентов, которые уже были на показе этого объекта/лота.
	•	Выход: список брокеров, которые показывают похожий проект, но не видели этот конкретный объект.

⸻

4) V3 — “умный мэтчинг” по содержанию (дорого, но максимальная ценность)

4.1 V3.A — “Мэтчинг через заметки/комментарии”

Почему:
	•	Сейчас ключевые детали живут в “комментариях справа” и не ищутся.
	•	В транскрипции: “читать примечания”.

Функция:
	•	Извлекать параметры спроса/интересов из текстовых полей и сопоставлять с объектом.

4.2 V3.B — “Мэтчинг через звонки (транскрипты)”

Почему:
	•	В транскрипции: “показы, звонки”, “мы расшифровываем звонки… в Москве не платили, но скоро будем”.
	•	Максимально близко к реальному запросу клиента.

Функция:
	•	Из транскриптов звонков вытаскивать критерии (ЖК, бюджет, метраж, этаж, отделка, район).
	•	Считать кандидатов “теплые” (кто уже ездит, на этапе показов/подбора).

4.3 V3.C — “Сегментация тепла”

Почему:
	•	В транскрипции: “можно показывать только этап показа — теплые”.

Функция:
	•	Фильтр “в работе / уже сделали N показов / активные”.

⸻

5) Интеграционный слой: где это живёт (параллельная ветка)

5.1 Локальный движок (то, что ты делаешь сейчас)
	•	CLI + JSON API.
	•	Источник данных: выгрузка/потом API.

5.2 Интерфейсы (после подтверждения V1)

Варианты, которые обсуждали:
	•	Telegram-бот (быстро как интерфейс)
	•	Окно в AmoCRM (сложнее: сущности собственник vs квартира)
	•	Вайтлист (сложно: “интегрировать CRM в Вайтлист”)

Приоритет:
	1.	Telegram/UI как “тонкий клиент” к локальному движку
	2.	Потом — нативная интеграция туда, где удобнее брокерам (Amo или WhiteList)

⸻

6) Границы продукта (из транскрипции — что НЕ делаем в ранних версиях)
	•	Не переезжаем на ReCRM: это референс идеи, не цель.
	•	Не строим полноценную новую CRM.
	•	Не делаем “идеальный мэтчинг” сразу: начинаем с того, что уже есть (показы).
	•	Не привязываемся к WhatsApp/Telegram-перепискам как источнику данных (там хаос и риски).

⸻

7) Метрики и “зачем бизнесу” (как обсуждали)
	•	Основная логика: рост показов → рост сделок.
	•	Для V1 достаточно доказательства полезности:
	•	брокеры реально используют список и пишут коллегам
	•	появляются дополнительные показы, инициированные подсказкой
